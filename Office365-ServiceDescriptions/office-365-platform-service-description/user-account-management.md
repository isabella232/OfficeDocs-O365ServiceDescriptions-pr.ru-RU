## <a name="sign-in-options"></a>Параметры входа в систему

Корпорация Майкрософт имеет две системы, которые можно использовать для удостоверений пользователей:
  
- **Учетная запись** рабочей или учебной учетной записи (облачное удостоверение) — пользователи получают Azure Active Directory облачные учетные данные (отдельно от других рабочих столов или корпоративных учетных данных) для входа в облачные службы Майкрософт. Это удостоверение по умолчанию, и его рекомендуется использовать, чтобы свести к минимуму сложность развертывания. Пароли для рабочих или учебных учетных записей используют Azure Active Directory [паролей](/previous-versions/azure/jj943764(v=azure.100)).
    
- **Федеративная** учетная запись (федеративное удостоверение) — для всех подписок в организациях с локальная служба Active Directory, использующих единый вход( единый вход), пользователи могут войти в службы Майкрософт с помощью своих учетных данных Active Directory. Корпоративная служба Active Directory хранит и контролирует политику паролей. Сведения о едином входе см. в стратегии [единого входа](/previous-versions/azure/azure-services/hh967643(v=azure.100)).
    
Тип удостоверения влияет на взаимодействие с пользователем и параметры управления учетными записями пользователей, а также на требования к оборудованию и программному обеспечению и другие аспекты развертывания.
  
### <a name="custom-domains-and-identity-options"></a>Личные домены и параметры удостоверений

При создании нового пользователя имя входа и адрес электронной почты пользователя назначаются домену по умолчанию, как задано в Центр администрирования Microsoft 365. Дополнительные сведения см. в [статье "Добавление пользователей и домена"](https://support.office.com/article/Add-your-users-and-domain-to-Office-365-6383f56d-3d09-4dcb-9b41-b5f5a5efd611). 
  
По умолчанию подписка использует домен *<.onmicrosoft.com*>, созданный с помощью учетной записи. Если вы используете Office 365 под управлением 21Vianet в Китае, доменом по умолчанию является <*companyname.onmsChina.cn*>. Вы можете добавить один или несколько личных доменов в корпорацию Майкрософт вместо сохранения  onmicrosoft.com домена и назначить пользователям возможность входа с помощью любого проверенного домена. Каждый назначенный домен пользователя  это электронный адрес, который будет появляться при отправке и получении электронных сообщений.
  
Вы можете разместить до 900 зарегистрированных доменов Интернета, каждый из которых представлен другим пространством имен. 
  
Для организаций, использующих единый вход, все пользователи в домене должны использовать одну и ту же систему идентификации: либо облачное, либо федеративное удостоверение. Например, у вас может быть одна группа пользователей, которым требуется только облачное удостоверение, так как они не имеют доступа к локальным системам, а другая группа пользователей, использующих Microsoft и локальные системы. Вы добавите два домена в Office 365, например contractors.contoso.com **и staff.contoso.com**,  и настроите единый вход только для одного из них. Весь домен можно конвертировать из облачного удостоверения в федеративное удостоверение или из федеративного удостоверения в облачное.
  
Дополнительные сведения о доменах в Office 365: см. в описании службы [Домены](domains.md). 
  
## <a name="authentication"></a>Проверка подлинности

За исключением сайтов Интернета для анонимного доступа, созданных с помощью SharePoint Online, пользователи должны быть прошедшими проверку подлинности при доступе к службы Майкрософт. 
  
- **Современная проверка подлинности** . Современная проверка подлинности обеспечивает вход на основе библиотеки проверки подлинности Майкрософт Office клиентских приложений на разных платформах. Это запускает такие возможности для входа, как многофакторная проверка подлинности (MFA), использование сторонних поставщиков удостоверений на основе SAML с клиентскими приложениями Office, а также проверки подлинности с помощью смарт-карт и на основе сертификатов. Это также устраняет необходимость использования Outlook microsoft. Дополнительные сведения, включая доступность современной проверки подлинности в приложениях Office, см. в статье о том, как работает современная проверка подлинности для клиентских приложений [Office 2013 и Office 2016](/office365/enterprise/modern-auth-for-office-2013-and-2016).
    
    Современная проверка подлинности включена по умолчанию для Exchange Online. Сведения о том, как включить или отключить ее, см. в статье "Включение современной проверки подлинности [в Exchange Online](/exchange/clients-and-mobile-in-exchange-online/enable-or-disable-modern-authentication-in-exchange-online)".
    
- **Проверка подлинности удостоверений в облаке** — пользователи с облачными удостоверениями выполняют проверку подлинности с помощью традиционных запросов и ответов. Веб-браузер перенаправляется в службу входа Майкрософт, где введите имя пользователя и пароль для рабочей или учебной учетной записи. Служба входа проверяет подлинность учетных данных, создает маркер службы, который веб-браузер отправляет в запрошенную службу, и выполняет вход пользователя. 
    
- **Федеративная** проверка подлинности на основе удостоверений — пользователи с федеративными удостоверениями выполняют проверку подлинности с помощью службы федерации Active Directory (AD FS) (AD FS) 2.0 или других служб маркеров безопасности. Веб-браузер перенаправляется в службу входа Майкрософт, где вы введите корпоративный идентификатор в форме имени участника-пользователя (UPN), например: *isabel@contoso.com*. Служба входа определяет, что вы являетесь частью федеративного домена, и предлагает перенаправить вас на локальный сервер федерации для аутентификации. При входе на рабочий стол (присоединенный к домену) выполняется проверка подлинности (с помощью Kerberos или NTLMv2), а локальная служба маркеров безопасности создает маркер входа, который веб-браузер отправляет в службу входа Майкрософт. С помощью этого маркера служба входа создает маркер, который веб-браузер отправляет в запрошенную службу, после чего выполняется вход. Список доступных служб маркеров безопасности см. в схеме [единого входа](/previous-versions/azure/azure-services/hh967643(v=azure.100)).
    
Корпорация Майкрософт использует проверку подлинности на основе форм, а трафик проверки подлинности по сети всегда шифруется с помощью TLS/SSL с помощью порта 443. Трафик проверки подлинности использует незначительный процент пропускной способности для службы Майкрософт. 
  
### <a name="multi-factor-authentication"></a>Многофакторная проверка подлинности

При многофакторной проверке подлинности пользователи должны подтвердить телефонный звонок, текстовое сообщение или уведомление приложения на смартфоне после правильного ввода пароля. Пользователь сможет войти только после прохождения второго этапа аутентификации. Администраторы Майкрософт могут регистрировать пользователей для многофакторной проверки подлинности в Центр администрирования Microsoft 365. Дополнительные сведения о [Многофакторной идентификации](/office365/admin/security-and-compliance/set-up-multi-factor-authentication).
  
### <a name="rich-client-authentication"></a>Проверка подлинности полнофункционального клиента

Для полнофункциональных клиентов, таких как классические приложения Microsoft Office, проверка подлинности может выполняться двумя способами.
  
- Помощник по Sign-In Microsoft **Online Services**. Помощник по входу, установленный при настройке рабочего стола, содержит клиентскую службу, которая получает маркер службы от службы входа и возвращает его расширенному клиенту. 
    
  - Если у вас есть облачное удостоверение, вы получите запрос на ввод учетных данных, которые клиентская служба отправляет службе входа для проверки подлинности (с помощью WS-Trust).
    
  - Если у вас есть федеративное удостоверение, служба клиента сначала обращается к серверу AD FS 2.0 для проверки подлинности учетных данных (с помощью Kerberos или NTLMv2) и получения маркера входа, который отправляется в службу входа (с помощью WS-Federation и WS-Trust).
    
- **Обычная или прокси-проверка подлинности по протоколу SSL**. Клиент Outlook передает базовые учетные данные проверки подлинности по протоколу SSL в Exchange Online. Exchange Online выполняет прокси-сервер проверки подлинности на платформе удостоверений, а затем локальная служба Active Directory сервера федерации (для единого входа). 
    
Чтобы обеспечить правильное обнаружение и проверку подлинности службы Майкрософт, администраторы должны применить набор компонентов и обновлений к каждой рабочей станции, использующей полнофункциональные клиенты (например, Microsoft Office 2010) и подключаться к Office 365. Настройка рабочего стола — это автоматизированное средство для настройки рабочих станций с необходимыми обновлениями. Дополнительные сведения см. в [статье "Использование текущего Office классических приложений"](https://support.office.com/article/3324b8b8-dceb-45e2-ac24-c642720108f7).

## <a name="delete-user-accounts"></a>Удаление учетных записей пользователей

Способ удаления учетных записей зависит от того, используется ли синхронизация каталогов. 
  
- Если синхронизация каталогов не используется, учетные записи можно удалить на странице администратора или с помощью Windows PowerShell.
    
- Если синхронизация каталогов используется, то необходимо удалять пользователей из локальных Active Directory, а не из Office 365:.
    
При удалении учетной записи она становится неактивной. В течение примерно 30 дней после удаления учетную запись можно восстановить. Дополнительные сведения об удалении и восстановлении учетных [](/office365/admin/add-users/delete-a-user) записей см[](/office365/admin/add-users/restore-user). в разделе "Удаление пользователей и восстановление пользователей". ## <a name="password-management"></a>Управление паролями

Политики и процедуры для управления паролями зависят от системы идентификации.
  
### <a name="cloud-identity-password-management"></a>Управление паролями для удостоверений в облаке
  
При использовании облачных удостоверений пароли создаются автоматически при создании учетной записи.
  
- Сведения о требованиях к надежности пароля облачного удостоверения см. в статье о [политике паролей](/previous-versions/azure/jj943764(v=azure.100)).
    
- Чтобы повысить безопасность, пользователи должны изменить свои пароли при первом доступе к службы Майкрософт. В результате, прежде чем пользователи смогут получить доступ к службы Майкрософт, они должны войти в Центр администрирования Microsoft 365, где им будет предложено изменить свои пароли.
    
- Администраторы могут задать политику истечения срока действия пароля. Дополнительные сведения см. в статье [Установка срока действия паролей](/office365/admin/manage/set-password-expiration-policy).
    
Существует несколько средств сброса паролей для пользователей с облачными удостоверениями:
  
- **Администратор сбрасывает** пароль. Если пользователи теряют или забудет свои пароли, администраторы могут сбросить пароли пользователей в Центре администрирования или с помощью Windows PowerShell. Пользователи могут изменять свои пароли, только если знают текущие пароли. 
    
    В корпоративных планах, если администраторы теряют или забудет свои пароли, другой администратор с ролью глобального администратора может сбросить пароли администраторов в Центр администрирования Microsoft 365 или с помощью Windows PowerShell. Дополнительные сведения см [. в разделе "Сброс паролей для администраторов"](/office365/admin/add-users/reset-passwords). Если вы работаете в Office 365 под управлением 21Vianet в Китае, см. сведения об изменении или сбросе паролей в Office 365 под управлением [21Vianet](https://support.office.com/article/change-or-reset-your-password-in-office-365-operated-by-21vianet-d8eb5b62-9d0e-4267-a9bf-2aa491ee6d0b).
    
- **Пользователь изменяет пароли** с помощью Outlook в Интернете— страница параметров Outlook в Интернете содержит гиперссылку "Изменить пароль", которая перенаправляет пользователей на страницу "Изменить **пароль**". Пользователь должен знать свой предыдущий пароль. Дополнительные сведения см. в разделе ["Изменение пароля"](https://support.office.com/article/change-password-in-outlook-web-app-50bb1309-6f53-4c24-8bfd-ed24ca9e872c). Если вы используете Office 365 под управлением 21Vianet в Китае, см. статью "Изменение или сброс паролей Office 365 под управлением [21Vianet"](https://support.office.com/article/change-or-reset-your-password-in-office-365-operated-by-21vianet-d8eb5b62-9d0e-4267-a9bf-2aa491ee6d0b).
    
- **Права** на сброс пароля на основе ролей. Для корпоративных планов полномочным пользователям, таким как сотрудники службы технической поддержки,  может быть назначено право пользователя сброса пароля и право на изменение паролей с помощью предопределенных или пользовательских ролей, не становясь полными администраторами служб. По умолчанию в корпоративных планах администраторы с ролью глобального администратора, администратора паролей или администратора управления пользователями могут изменять пароли. Дополнительные сведения см. в [разделе "Назначение ролей администратора"](/office365/admin/add-users/assign-admin-roles).
    
- **Сброс паролей с Windows PowerShell** — администраторы служб могут использовать Windows PowerShell для сброса паролей. 
    
### <a name="federated-identity-password-management"></a>Управление паролями федеративных удостоверений
  
При использовании федеративных удостоверений управление паролями происходит в Active Directory. Локальная служба маркеров безопасности согласовывает проверку подлинности со шлюзом федерации, не передавая локальные пароли Active Directory пользователей через Интернет в Office 365. Используются локальные политики паролей или в случае веб-клиентов двухфакторная идентификация. Outlook в Интернете не включает гиперссылку "Изменить пароль". Пользователи изменяют пароли с помощью стандартных локальных средств или с помощью параметров входа своих настольных ПК.
  
Если в вашей среде организации включена синхронизация каталогов с единым входом [(SSO)](/previous-versions/azure/azure-services/dn441213(v=azure.100)) и имеется сбой, который влияет на федеративного поставщика удостоверений, резервное копирование синхронизации паролей для федеративного входа позволяет вручную переключить домен на синхронизацию паролей. Синхронизация паролей позволит пользователям получать доступ, пока сбой устранен. Узнайте [, как переключиться с одного Sign-On на синхронизацию паролей](https://go.microsoft.com/fwlink/p/?LinkId=509832).
  
## <a name="license-management"></a>Управление лицензиями

Лицензия предоставляет пользователю доступ к набору службы Майкрософт. Администратор назначает лицензию каждому пользователю для службы, доступ к которой ему требуется. Например, можно назначить пользователю доступ к Skype для бизнеса Online, но не к SharePoint Online.
  
Администраторы выставления счетов Майкрософт могут вносить изменения в сведения о подписке, такие как количество пользовательских лицензий и количество дополнительных служб, которые использует ваша компания. [Назначьте или удалите лицензию](/office365/admin/subscriptions-and-billing/assign-licenses-to-users). Если вы используете Office 365 21Vianet, см. статью "Назначение или удаление лицензий в Office 365 под управлением [21Vianet"](/office365/admin/subscriptions-and-billing/assign-licenses-to-users).
  
## <a name="group-management"></a>Управление группами

Группы безопасности используются в SharePoint Online для управления доступом к сайтам. Группы безопасности можно создать в Центр администрирования Microsoft 365. Дополнительные сведения о группах безопасности см. в разделе ["Создание, изменение или удаление группы безопасности"](/office365/admin/email/create-edit-or-delete-a-security-group).
  
## <a name="administrator-roles"></a>Роли администратора

Office 365 для предприятий следует модели управления доступом на основе ролей (RBAC): разрешения и возможности определяются ролями управления. Пользователь, зарегистрировавший Office 365 в своей организации, автоматически становится глобальным администратором или администратором верхнего уровня. Существует пять ролей администратора: глобальный администратор, администратор выставления счетов, администратор паролей, администратор служб и администратор управления пользователями. Дополнительные сведения о ролях администраторов в Office 365 для предприятий, в том числе о том, как они применяются к Exchange Online, SharePoint Online и администрированию Skype для бизнеса Online, см. в разделе "Назначение ролей администратора[".](/previous-versions/windows/it-pro/windows-server-2012-R2-and-2012/jj878348(v=ws.11)) Если вы используете Office 365 под управлением 21Vianet в Китае, см. статью "Назначение ролей администратора Office 365 [для бизнеса"](/office365/admin/add-users/assign-admin-roles).
  
## <a name="delegated-administration-and-support-for-partners"></a>Делегированное администрирование и поддержка партнеров

Партнеры могут быть уполномочены управлять учетными записями от имени клиентов. Клиенту не требуется учетная запись пользователя для партнеров, и он не использует лицензию при предоставлении делегированного центра администрирования. Партнерам можно назначить полный или ограниченный доступ к пользователям организации. Ограниченный доступ включает права на сброс паролей, управление запросами служб и наблюдение за работоспособностью системы. 
  
> [!NOTE]
> Указывать партнера в качестве полномочного администратора можно не во всех регионах.
